# Core：发布 v0.4.1

## 变更背景 / 目标
- `master` 已包含 Windows RFCOMM listener 的稳定性修复；
- 需要发布一个补丁版本，给 `MyFlowHub-Win` 提供可引用的稳定 Core 版本；
- 本次不新增实现逻辑，只做发布审计、版本锚点与归档。

## 具体变更内容
### 新增
- `plan.md`
  - 记录本轮 `v0.4.1` 发布 workflow 的目标、任务、验证、回滚点。
- `docs/change/2026-03-14_core-v0.4.1-release.md`
  - 归档本次补丁版本发布的背景、范围和验证证据。

### 发布动作
- 基于 `master` 当前已验证提交创建 `v0.4.1` annotated tag；
- `v0.4.1` 对应内容为 Windows RFCOMM listener 修复补丁，不夹带新功能。

## 对应 plan.md 任务映射
- `CORE-REL-1`：审计发布基线
- `CORE-REL-2`：维护发布计划与归档
- `CORE-REL-3`：执行 tag 与远端发布准备
- `CORE-REL-4`：Code Review 与收敛

## 关键设计决策与权衡
- **补丁发布，不放大改动面**
  - 只发布已经在 `master` 中验证过的修复，避免再引入行为变化。
- **先固定版本锚点，再让下游升级**
  - Win 侧依赖升级应指向稳定 tag，而不是临时分支或未发布提交。
- **文档独立归档**
  - 版本发布单独成文，避免与实现修复文档混淆。

## 测试与验证方式 / 结果
- 基线验证：
  - `GOWORK=off go test ./... -count=1`：已通过
- 发布验证：
  - 确认 `master` 最新提交为 Windows RFCOMM listener 修复合并结果；
  - 确认 `v0.4.0` 之后尚无新 tag，占位 `v0.4.1` 合理；
  - 已在提交 `373a816` 上创建并推送 annotated tag：`v0.4.1`。

## 潜在影响与回滚方案
- 潜在影响：
  - 下游仓库将开始引用 `v0.4.1`，因此该 tag 会成为 Windows RFCOMM 修复的公开版本锚点。
- 回滚方案：
  - 若发现发布错误，可删除本地/远端 `v0.4.1` tag；
  - 下游仓库可回退到 `v0.4.0` 或改用后续修复版本。
